# Syntavra Token-Saver Product Unification Plan

Syntavra remains **0.0.1 / pre-release**. This plan converts the broad internal runtime into one measurable token/context optimization product without fabricating external evidence.

## Product target

> A local-first skill and middleware layer that reduces coding-agent MCP schemas, repository context, tool output and session-memory overhead while preserving exact recovery and measuring provider-observed cost per verified successful task.

## Integrated workstreams

### P0 — Product identity and one control plane

- Position Syntavra as a token/context optimization skill, not a replacement coding agent.
- Keep `setup / status / run / prove` as the public mental model.
- Derive all MCP profile visibility from one canonical registry.
- Keep `tiny / optimized / full` as compatibility aliases only.
- Execute MCP calls through one explicit authorization and output pipeline.
- Remove active documentation references to deleted branches or unfinished security work.

### P1 — Token attribution and saving engines

- Record provider-observed fresh, cached, output and reasoning usage in the existing hash chain.
- Record source-level attribution separately for system, skill, schema, repository, tool-output, memory, conversation and response tokens.
- Label every attribution value as provider-observed, locally tokenized, estimated or unknown.
- Compile MCP discovery schemas deterministically and preserve readable argument aliases.
- Use command-specific compactors for common Git, test, lint, container, Kubernetes and GitHub CLI output.
- Build mandatory/likely/optional task context packs from exact definitions, impact, tests and verifiers.
- Preserve every omitted path or output behind exact recovery metadata.

### P1 — Primary host certification

- Codex, Claude Code and Cursor are the first certification targets.
- Each target requires clean installation, actual skill/MCP activation, real provider usage, verifier success, restart continuity and exact rollback on at least two operating systems.
- Other hosts remain contract-tested or declared until equivalent receipts exist.

### P1 — External competitor benchmark

SignalBench arms:

1. plain host;
2. Caveman;
3. RTK;
4. Token Savior;
5. repository-context competitor;
6. combined competitor pack;
7. Syntavra minimal;
8. Syntavra balanced.

All arms must use the same frozen repository, prompt, model, reasoning mode, context window, verifier, permissions, timeout, cache policy and hardware class. A missing command, provider receipt or verifier result fails the arm.

### P2 — Distribution

- npm installer with provenance under the `next` tag;
- PyPI package through trusted publishing;
- checksum and attestation backed portable binaries;
- SBOM, deterministic manifest and pre-release GitHub artifacts;
- post-install `syntavra status --doctor` verification.

## Release boundaries

Repository code can complete the architecture, tests, adapters and release automation. It cannot manufacture:

- provider billing receipts;
- competitor executions;
- independent reproductions;
- public registry credentials;
- real users;
- 90 days of operational history.

Those remain explicit external gates and do not justify changing the version.
